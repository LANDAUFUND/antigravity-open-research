# README АРХИВНОГО ПАКЕТА АНТИГРАВИТАЦИИ

АНТИГРАВИТАЦИЯ РАССМАТРИВАЕТСЯ КАК ИССЛЕДОВАТЕЛЬСКОЕ НАПРАВЛЕНИЕ, ТРЕБУЮЩЕЕ МАТЕМАТИЧЕСКОЙ ФОРМАЛИЗАЦИИ, ЭКСПЕРИМЕНТАЛЬНОЙ ПРОВЕРКИ, ИЗМЕРИМОСТИ И ВОСПРОИЗВОДИМОСТИ.

## 1. НАЗНАЧЕНИЕ РАЗДЕЛА

Этот README задает публичную структуру архивного пакета для DOI/архива. Он описывает, какие файлы можно публиковать, какие файлы должны оставаться во внутреннем контуре, как читать агрегированные данные и почему архивная публикация не является доказательством антигравитации.

## 2. БАЗОВЫЕ ФОРМУЛЫ

Проверка целостности:

```text
SHA256(file) = hash
```

Идентификатор версии:

```text
VERSION = MAJOR.MINOR.PATCH
```

Относительное изменение веса:

```text
η = ΔP / P0
P0 = m · g
z = ΔP / uΔ
```

## 3. РАСШИФРОВКА ПЕРЕМЕННЫХ

- `file` - файл архивного пакета.
- `hash` - SHA-256 контрольная сумма файла.
- `VERSION` - версия архивного пакета.
- `MAJOR` - изменение структуры пакета или публичного протокола.
- `MINOR` - добавление новых таблиц, полей или отчетов.
- `PATCH` - исправление опечаток, ссылок, пояснений без изменения данных.
- `η` - относительное изменение веса.
- `ΔP` - изменение веса.
- `P0` - контрольный вес.
- `m` - масса.
- `g` - ускорение свободного падения.
- `z` - нормированное отклонение.
- `uΔ` - неопределенность отклонения.

## 4. ЕДИНИЦЫ ИЗМЕРЕНИЯ

- Хэш: шестнадцатеричная строка.
- Версия: текстовая строка формата `MAJOR.MINOR.PATCH`.
- Масса: `кг`.
- Сила / вес: `Н`.
- Ускорение: `м/с²`.
- Относительные величины: доля, `ppm` или безразмерный индекс.

## 5. ПРИМЕРЫ РАСЧЕТОВ

Если файл `РЕЕСТР_МЕТААНАЛИЗА_СЕРИЙ_АНТИГРАВИТАЦИИ.csv` имеет SHA-256:

```text
5b816f2c7f5616497d16b20dfbcc96775ee535c55393e6220fe43ee8d6ae7b3d
```

то эта строка должна совпадать в локальной папке, GitHub-релизе и DOI/архиве.

Если `ΔP = 0.00001 Н`, `P0 = 1.96133 Н`, то:

```text
η = 0.00001 / 1.96133 ≈ 5.1 ppm
```

Без неопределенности и контрольных серий это число не является доказательством эффекта.

## 6. ЧТО ИМЕННО ДОЛЖНА ПРОВЕРИТЬ ЛАБОРАТОРИЯ

- Все ли опубликованные файлы перечислены в манифесте.
- Есть ли SHA-256 для каждого файла.
- Совпадают ли хэши локальной, GitHub и архивной копии.
- Нет ли raw data, персональных данных, закрытых параметров стендов или патентных деталей.
- Есть ли ссылка на исходный GitHub commit.
- Указаны ли лицензия, авторство, ограничение интерпретации и дата выпуска.

## 7. КАКИЕ ДАННЫЕ НУЖНО СОБРАТЬ

- Название пакета.
- Версия пакета.
- GitHub commit.
- DOI или архивный URL после публикации.
- Список файлов.
- Размеры файлов.
- SHA-256 хэши.
- Лицензия.
- Ответственный review.
- Статус: черновик, готов к release review, опубликован, заменен новой версией.

## 8. КАКИЕ ВЫВОДЫ ДЕЛАТЬ НЕЛЬЗЯ

- Нельзя считать DOI доказательством антигравитации.
- Нельзя публиковать закрытые лабораторные параметры как публичные агрегаты.
- Нельзя менять опубликованный архив без новой версии.
- Нельзя удалять отрицательные или нулевые серии ради красивого вывода.
- Нельзя использовать архив без ссылки на ограничения и методику.

## 9. КАК ОТЛИЧАТЬ НАУЧНЫЙ РАСЧЕТ ОТ ПСЕВДОНАУЧНОГО ЗАЯВЛЕНИЯ

Научный архив имеет версию, манифест, хэши, словарь данных, методику, отрицательные результаты и ограничения. Псевдонаучный архив имеет громкое название, но не позволяет проверить происхождение данных, хэши, единицы, неопределенность и критерии исключения артефактов.

## 10. КАКИЕ СЛЕДУЮЩИЕ ДОКУМЕНТЫ НУЖНО ПОДГОТОВИТЬ

- Фактический release candidate для первого DOI/архива.
- Графический индекс серий и параметров.
- Публичный протокол цитирования.
- Сводку изменений между версиями архива.
- Отдельный юридический review лицензии и donation/funding-политики.
