# ПРОТОКОЛ ВЕРСИОНИРОВАНИЯ И ХЭШЕЙ АРХИВА АНТИГРАВИТАЦИИ

АНТИГРАВИТАЦИЯ РАССМАТРИВАЕТСЯ КАК ИССЛЕДОВАТЕЛЬСКОЕ НАПРАВЛЕНИЕ, ТРЕБУЮЩЕЕ МАТЕМАТИЧЕСКОЙ ФОРМАЛИЗАЦИИ, ЭКСПЕРИМЕНТАЛЬНОЙ ПРОВЕРКИ, ИЗМЕРИМОСТИ И ВОСПРОИЗВОДИМОСТИ.

## 1. НАЗНАЧЕНИЕ РАЗДЕЛА

Протокол задает правила версий и контрольных сумм для архивных пакетов. Он нужен, чтобы GitHub, Bitrix24 и DOI/архив ссылались на одну и ту же проверяемую версию.

## 2. БАЗОВЫЕ ФОРМУЛЫ

```text
SHA256(file) = hash
PACKAGE_HASH = SHA256(sorted_manifest)
VERSION = MAJOR.MINOR.PATCH
```

Связь агрегата с версией:

```text
ARCHIVE_RECORD = ID_АГРЕГАТА + VERSION + SHA256(file)
```

## 3. РАСШИФРОВКА ПЕРЕМЕННЫХ

- `file` - отдельный файл пакета.
- `hash` - SHA-256 файла.
- `sorted_manifest` - манифест, отсортированный по имени файла.
- `PACKAGE_HASH` - контрольная сумма всего пакета через манифест.
- `VERSION` - версия пакета.
- `MAJOR` - несовместимое изменение структуры или методики.
- `MINOR` - добавление совместимых данных или документов.
- `PATCH` - исправление без изменения данных.
- `ARCHIVE_RECORD` - строка связи агрегата, версии и файла.

## 4. ЕДИНИЦЫ ИЗМЕРЕНИЯ

- Хэш: полная SHA-256 строка.
- Версия: текст `MAJOR.MINOR.PATCH`.
- Размер файла: байты.
- Дата сборки: ISO 8601.

## 5. ПРИМЕРЫ РАСЧЕТОВ

Если исправлена только опечатка в README, версия меняется с `1.0.0` на `1.0.1`.

Если добавлен новый CSV-реестр, версия меняется с `1.0.1` на `1.1.0`.

Если изменена структура публичных полей, версия меняется с `1.1.0` на `2.0.0`.

## 6. ЧТО ИМЕННО ДОЛЖНА ПРОВЕРИТЬ ЛАБОРАТОРИЯ

- Все ли файлы пакета перечислены в манифесте.
- Все ли хэши пересчитаны после финальной сборки.
- Совпадает ли GitHub commit с манифестом.
- Есть ли статус `REPLACED`, если архивная версия заменена.
- Нельзя ли спутать internal release candidate и public release.

## 7. КАКИЕ ДАННЫЕ НУЖНО СОБРАТЬ

- Имя файла.
- Путь в пакете.
- Размер в байтах.
- SHA-256.
- Тип файла.
- Публичный статус.
- GitHub commit.
- Bitrix24 file ID, если есть.
- DOI/архивный URL, если есть.
- Версия пакета.

## 8. КАКИЕ ВЫВОДЫ ДЕЛАТЬ НЕЛЬЗЯ

- Нельзя считать две версии одинаковыми, если хотя бы один хэш отличается.
- Нельзя менять файл после публикации без новой версии.
- Нельзя указывать короткий хэш вместо полного SHA-256 в архивном манифесте.
- Нельзя считать манифест научной валидацией результата.

## 9. КАК ОТЛИЧАТЬ НАУЧНЫЙ РАСЧЕТ ОТ ПСЕВДОНАУЧНОГО ЗАЯВЛЕНИЯ

Научная публикация допускает проверку цепочки "файл - хэш - commit - архив". Псевдонаучная публикация ссылается на документы, которые изменяются без версий и контрольных сумм.

## 10. КАКИЕ СЛЕДУЮЩИЕ ДОКУМЕНТЫ НУЖНО ПОДГОТОВИТЬ

- Скрипт проверки манифеста.
- Release notes для версии `1.0.0`.
- Политику замены версии.
- Таблицу соответствия Bitrix24 file IDs и GitHub paths.
- Инструкцию внешней проверки архивного пакета.
