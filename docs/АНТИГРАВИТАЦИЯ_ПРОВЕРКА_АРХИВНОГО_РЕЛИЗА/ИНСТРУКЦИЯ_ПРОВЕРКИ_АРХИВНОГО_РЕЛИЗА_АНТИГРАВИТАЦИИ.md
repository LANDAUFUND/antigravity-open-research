# ИНСТРУКЦИЯ ПРОВЕРКИ АРХИВНОГО РЕЛИЗА АНТИГРАВИТАЦИИ

Версия: `0.1.0`.

Статус: рабочий контрольный слой перед публичным архивом и DOI.

Каноническая формулировка:

`АНТИГРАВИТАЦИЯ РАССМАТРИВАЕТСЯ КАК ИССЛЕДОВАТЕЛЬСКОЕ НАПРАВЛЕНИЕ, ТРЕБУЮЩЕЕ МАТЕМАТИЧЕСКОЙ ФОРМАЛИЗАЦИИ, ЭКСПЕРИМЕНТАЛЬНОЙ ПРОВЕРКИ, ИЗМЕРИМОСТИ И ВОСПРОИЗВОДИМОСТИ`.

Эта инструкция не доказывает антигравитацию. Она задает порядок проверки того, что архивный пакет технически готов к review, воспроизводимому цитированию и последующей DOI-публикации.

## 1. Назначение раздела

Раздел нужен для превращения набора документов в проверяемый архивный релиз. Проверяются не физические утверждения сами по себе, а целостность пакета: список файлов, размеры, SHA-256, соответствие Bitrix24 file ID, соответствие GitHub paths, статус публичности, лицензия и отсутствие незакрытых служебных полей.

## 2. Входные артефакты

Минимальный входной набор:

- `МАНИФЕСТ_DOI_АРХИВА_АНТИГРАВИТАЦИИ.csv`;
- локальная папка архивного пакета;
- SHA-256 каждого файла;
- таблица соответствия Bitrix24 и GitHub;
- публичный GitHub commit для released/public-candidate версии;
- закрытый GitHub commit для внутренней копии;
- статус review: `DRAFT`, `PUBLIC_CANDIDATE`, `RELEASE_CANDIDATE`, `RELEASED`.

## 3. Базовые формулы и проверки

Проверка целостности файла:

```text
SHA256(local_file) = SHA256(manifest_record)
```

Проверка размера:

```text
size_bytes(local_file) = РАЗМЕР_БАЙТ(manifest_record)
```

Проверка полноты манифеста:

```text
N_manifest_rows = N_files_in_release_scope
```

Проверка статуса:

```text
PUBLIC_RELEASE_ALLOWED = no_secrets AND no_raw_sensitive_data AND no_unreviewed_breakthrough_claims AND all_hashes_verified
```

## 4. Расшифровка переменных

- `SHA256(local_file)` - хэш локального файла по алгоритму SHA-256.
- `SHA256(manifest_record)` - значение из колонки `SHA256`.
- `size_bytes(local_file)` - фактический размер файла в байтах.
- `N_manifest_rows` - число строк данных в манифесте без заголовка.
- `N_files_in_release_scope` - число файлов, которые входят в текущий архивный релиз.
- `PUBLIC_RELEASE_ALLOWED` - логический флаг, показывающий, что пакет можно передавать на финальное научное и юридическое review.

## 5. Единицы измерения

- Размер файла: байт.
- Хэш: шестнадцатеричная строка длиной 64 символа.
- Версия пакета: строка SemVer вида `MAJOR.MINOR.PATCH`.
- GitHub commit: короткий или полный хэш commit.
- Bitrix24 file ID: числовой идентификатор объекта файла.

## 6. Пример проверки

Пример для файла README:

```text
ИМЯ_ФАЙЛА = README_АРХИВНОГО_ПАКЕТА_АНТИГРАВИТАЦИИ.md
BITRIX24_FILE_ID = 420282
PUBLIC_GITHUB_COMMIT = 477b407
SHA256 = 62620cc0ca9bd92eea8ec29e7ef186195c1751809e8e11fc0d4b25e5f370bac1
```

Если локальный хэш совпадает с манифестом и скачанная из Bitrix24 копия дает тот же SHA-256, файл считается технически зафиксированным. Это не означает, что научное содержание доказано.

## 7. Что именно должна проверить лаборатория

- Все файлы из релизного списка существуют локально.
- Каждый файл имеет рассчитанный размер и SHA-256.
- В манифесте не осталось значений `ТРЕБУЕТ_РАСЧЕТА`, `ТРЕБУЕТ_SHA256`, `ТРЕБУЕТ_COMMIT`, `ТРЕБУЕТ_BITRIX_ID`.
- Bitrix24 file ID указывает на тот же файл, который указан в GitHub и локальном пакете.
- Публичные файлы не содержат секретов, персональных данных, закрытых параметров стендов или заявлений о доказанной антигравитации.
- Release notes отделяют методический пакет от экспериментального результата.

## 8. Какие данные нужно собрать

- Полный список файлов релиза.
- Локальные пути файлов.
- GitHub paths.
- Bitrix24 folder ID и file IDs.
- SHA-256 каждого файла.
- Размер каждого файла в байтах.
- Версию пакета.
- Статус public/private/release candidate.
- Ответственного за научный review.
- Ответственного за проверку лицензии.
- Дату финальной проверки.

## 9. Какие выводы делать нельзя

- Нельзя писать, что антигравитация доказана.
- Нельзя выводить физический эффект из факта публикации архива.
- Нельзя заменять независимую репликацию наличием DOI.
- Нельзя считать совпадение хэшей подтверждением научной гипотезы.
- Нельзя публиковать raw data, чувствительные параметры стендов или патентоспособные детали без отдельного review.

## 10. Как отличать научный расчет от псевдонаучного заявления

Научный расчет:

- указывает модель, формулы, единицы, пределы применимости и измеримые переменные;
- хранит исходные данные или публично объясняет, почему они закрыты;
- допускает отрицательный результат;
- допускает независимую репликацию;
- не делает выводов сильнее, чем позволяют данные.

Псевдонаучное заявление:

- ссылается на секретность вместо воспроизводимых данных;
- использует DOI или GitHub как замену эксперименту;
- смешивает UAP/НЛО, гипотезы и доказанные эффекты;
- не задает контрольные режимы;
- не допускает фальсификацию.
