- `АНТИГРАВИТАЦИЯ_МАШИННЫЕ_ДОКАЗАТЕЛЬСТВА_REVIEW_1_0_0`

## 1. НАЗНАЧЕНИЕ

Документ фиксирует порядок назначения людей или организаций на review-роли перед возможным финальным выпуском `RELEASE_1_0_0`.

Этот документ не утверждает, что review завершено. Он не объявляет `GO`. Он переводит текущий статус из абстрактного `ТРЕБУЕТ НАЗНАЧЕНИЯ` в управляемую процедуру назначения, проверки независимости, выдачи задания и получения письменного заключения.

## 2. ТЕКУЩЕЕ РЕШЕНИЕ ДО НАЗНАЧЕНИЯ

До фактического назначения рецензентов и закрытия их письменных заключений действует статус:

```text
RELEASE_1_0_0 = HOLD / NO-GO ПО УМОЛЧАНИЮ
```

Это не означает отказ от релиза. Это означает, что релиз нельзя объявлять, пока не закрыты блокирующие несоответствия `NC-001..NC-006`.

## 3. РОЛИ, КОТОРЫЕ ДОЛЖНЫ БЫТЬ НАЗНАЧЕНЫ

Минимальный набор:

| Код | Роль | Закрывает | Блокирует релиз |
|---|---|---|---|
| `SCI` | Научный рецензент | `NC-001` | Да |
| `DATA` | Ответственный за данные и воспроизводимость | `NC-002` | Да |
| `LIC` | Лицензионный проверяющий | `NC-003` | Да |
| `PAT` | Патентный проверяющий | `NC-004` | Да |
| `SEC` | Проверяющий безопасности и приватности | `NC-005` | Да |
| `REL` | Release manager | `NC-006` | Да |
| `ARC` | Архивариус / хранитель хэшей | Поддерживает `NC-002`, `NC-006` | Да, если хэши не подтверждены |

## 4. ПРАВИЛО НАЗНАЧЕНИЯ

Роль считается назначенной только если есть запись:

```text
РОЛЬ:
ФИО / ОРГАНИЗАЦИЯ:
КОНТАКТНЫЙ КАНАЛ:
ДАТА НАЗНАЧЕНИЯ:
ОБЪЕМ ПРОВЕРКИ:
КОНФЛИКТ ИНТЕРЕСОВ:
ОГРАНИЧЕНИЯ НЕЗАВИСИМОСТИ:
СРОК ЗАКЛЮЧЕНИЯ:
ОЖИДАЕМЫЙ ВЫХОД:
СТАТУС:
```

Если хотя бы одно обязательное поле не заполнено, роль остается `НЕ НАЗНАЧЕНО`.

## 5. КРИТЕРИЙ НЕЗАВИСИМОСТИ

Review не считается независимым, если проверяющий:

- проверяет собственный текст без внешней сверки;
- не имеет права поставить `NO-GO`;
- зависит от результата финансово или организационно без раскрытия конфликта;
- не получил исходный scope проверки;
- не видел список недопустимых выводов;
- не подтвердил границу между публичным и закрытым контуром.

Если независимость ограничена, это не запрещает участие автоматически, но требует явной записи в реестре.

## 6. ЧТО ДОЛЖЕН ПОЛУЧИТЬ КАЖДЫЙ REVIEWER

Минимальный пакет:

1. Назначение роли и scope.
2. Список входных артефактов.
3. Критерии проверки.
4. Список блокирующих рисков.
5. Форму заключения.
6. Журнал несоответствий `NC-001..NC-006`.
7. Машинные доказательства хэшей и версий.
8. Прямое указание: можно поставить `NO-GO`.

## 7. ЧТО ЗАПРЕЩЕНО

Запрещено:

- закрывать `NC` без назначенного ответственного;
- закрывать `NC` устно;
- закрывать `NC` фразой `проверено`;
- объявлять `RELEASE_1_0_0` из-за наличия GitHub-коммита;
- считать Bitrix24-загрузку научной проверкой;
- считать машинный SHA-256 человеческим review;
- публиковать патентно чувствительный материал как public без patent review;
- писать, что антигравитация доказана.

## 8. ВЫХОД ЭТОГО ЭТАПА

После завершения этапа должны быть заполнены:

- реестр назначений review-ролей;
- план закрытия `NC-001..NC-006`;
- таблица закрытия `NC-001..NC-006`;
- пакет заданий рецензентам;
- временное решение `NO-GO / HOLD` до фактического review.

Финальный `GO` возможен только в следующем этапе.
