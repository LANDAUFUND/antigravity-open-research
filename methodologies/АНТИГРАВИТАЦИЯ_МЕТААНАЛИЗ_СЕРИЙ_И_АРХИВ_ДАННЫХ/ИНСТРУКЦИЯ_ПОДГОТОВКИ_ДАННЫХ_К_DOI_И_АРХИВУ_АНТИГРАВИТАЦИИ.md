# ИНСТРУКЦИЯ ПОДГОТОВКИ ДАННЫХ К DOI И АРХИВУ АНТИГРАВИТАЦИИ

АНТИГРАВИТАЦИЯ РАССМАТРИВАЕТСЯ КАК ИССЛЕДОВАТЕЛЬСКОЕ НАПРАВЛЕНИЕ, ТРЕБУЮЩЕЕ МАТЕМАТИЧЕСКОЙ ФОРМАЛИЗАЦИИ, ЭКСПЕРИМЕНТАЛЬНОЙ ПРОВЕРКИ, ИЗМЕРИМОСТИ И ВОСПРОИЗВОДИМОСТИ.

## 1. НАЗНАЧЕНИЕ РАЗДЕЛА

Документ задает порядок подготовки публичного пакета данных к DOI/архиву: например Zenodo, Open Science Framework или институциональному архиву. Цель - сделать опубликованные агрегаты цитируемыми, проверяемыми и безопасными для public release.

## 2. БАЗОВЫЕ ФОРМУЛЫ

Контроль целостности файла:

```text
SHA256(file) = hash
```

Относительная неопределенность:

```text
ur = u / |x|
```

Сводный эффект серии:

```text
η = ΔP / P0
z = ΔP / uΔ
```

Порог публикации агрегата:

```text
nseries ≥ nmin
```

Ожидаемый вес:

```text
P0 = m · g
```

## 3. РАСШИФРОВКА ПЕРЕМЕННЫХ

- `SHA256(file)` - криптографический хэш файла.
- `hash` - строка контрольной суммы.
- `ur` - относительная неопределенность.
- `u` - стандартная неопределенность величины.
- `x` - измеряемая величина.
- `η` - относительное изменение веса.
- `ΔP` - изменение веса.
- `P0` - контрольный вес.
- `z` - нормированное отклонение.
- `nseries` - число серий в публичном агрегате.
- `nmin` - минимальное число серий для выбранного типа публикации.

## 4. ЕДИНИЦЫ ИЗМЕРЕНИЯ

- Хэш: текстовая строка в шестнадцатеричном формате.
- `ur`, `η`, `z`: безразмерные величины.
- `u`, `x`, `ΔP`, `P0`: единицы исходной физической величины, чаще `Н`, `кг`, `К`, `Па`, `Тл`, `В/м`.
- `nseries`, `nmin`: количество серий.

## 5. ПРИМЕРЫ РАСЧЕТОВ

Если публичный агрегат содержит `nseries = 12`, а минимальное правило публикации для нулевой карты `nmin = 5`, то пакет можно готовить к архиву при условии прохождения review.

Если `ΔP = 0.00001 Н`, `uΔ = 0.00003 Н`, то:

```text
z = 0.00001 / 0.00003 ≈ 0.33
```

Такой агрегат публикуется как результат, совместимый с нулем, а не как слабый положительный эффект.

## 6. ЧТО ИМЕННО ДОЛЖНА ПРОВЕРИТЬ ЛАБОРАТОРИЯ

- Нет ли в файлах персональных данных, секретов, закрытых параметров стендов или патентоспособных деталей.
- Все ли единицы указаны явно.
- Все ли агрегаты воспроизводимы из внутреннего набора данных.
- Есть ли README, словарь данных, лицензия, авторство, методика и список файлов.
- Совпадают ли SHA-256 хэши после выгрузки.
- Указано ли, что публикация не доказывает антигравитацию.

## 7. КАКИЕ ДАННЫЕ НУЖНО СОБРАТЬ

- Публичные агрегированные таблицы.
- Словарь полей.
- Методика получения агрегатов.
- Список исключенных закрытых полей с причиной исключения.
- Хэши всех файлов.
- Версия GitHub-коммита, из которого собран пакет.
- Дата публикации, авторство, лицензия, DOI или архивный URL после выпуска.

## 8. КАКИЕ ВЫВОДЫ ДЕЛАТЬ НЕЛЬЗЯ

- Нельзя публиковать raw data, если она не прошла review.
- Нельзя выдавать DOI за научное доказательство результата.
- Нельзя публиковать закрытые режимы стендов, если они находятся в патентном или защищенном контуре.
- Нельзя изменять данные после получения DOI без новой версии архива.
- Нельзя скрывать отрицательные и нулевые серии из публичного агрегата.

## 9. КАК ОТЛИЧАТЬ НАУЧНЫЙ РАСЧЕТ ОТ ПСЕВДОНАУЧНОГО ЗАЯВЛЕНИЯ

Научная архивная публикация содержит версию, хэши, методику, словарь полей, ограничения и отрицательные результаты. Псевдонаучная публикация пытается заменить эти элементы громким названием, DOI без воспроизводимости или ссылкой на закрытые данные без проверяемой структуры.

## 10. КАКИЕ СЛЕДУЮЩИЕ ДОКУМЕНТЫ НУЖНО ПОДГОТОВИТЬ

- `README` публичного пакета данных.
- `DATA_DICTIONARY` для агрегатов.
- `LICENSE_NOTICE` для DOI/архива.
- `RELEASE_REVIEW_CHECKLIST`.
- `ARCHIVE_MANIFEST` со списком файлов и SHA-256.
