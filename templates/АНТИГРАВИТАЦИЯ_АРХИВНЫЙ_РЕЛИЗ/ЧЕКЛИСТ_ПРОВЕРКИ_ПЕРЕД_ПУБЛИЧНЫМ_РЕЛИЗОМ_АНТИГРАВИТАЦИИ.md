# ЧЕКЛИСТ ПРОВЕРКИ ПЕРЕД ПУБЛИЧНЫМ РЕЛИЗОМ АНТИГРАВИТАЦИИ

АНТИГРАВИТАЦИЯ РАССМАТРИВАЕТСЯ КАК ИССЛЕДОВАТЕЛЬСКОЕ НАПРАВЛЕНИЕ, ТРЕБУЮЩЕЕ МАТЕМАТИЧЕСКОЙ ФОРМАЛИЗАЦИИ, ЭКСПЕРИМЕНТАЛЬНОЙ ПРОВЕРКИ, ИЗМЕРИМОСТИ И ВОСПРОИЗВОДИМОСТИ.

## 1. НАЗНАЧЕНИЕ РАЗДЕЛА

Чеклист используется перед публикацией материалов в GitHub, Bitrix24-публичный контур, DOI/архив или внешний отчет. Он должен отсечь секреты, raw data, патентные риски, псевдонаучные формулировки и непроверенные выводы.

## 2. БАЗОВЫЕ ФОРМУЛЫ

```text
P0 = m · g
η = ΔP / P0
z = ΔP / uΔ
SHA256(file) = hash
```

## 3. РАСШИФРОВКА ПЕРЕМЕННЫХ

- `P0` - контрольный вес.
- `m` - масса.
- `g` - ускорение свободного падения.
- `η` - относительное изменение веса.
- `ΔP` - изменение веса.
- `uΔ` - неопределенность изменения веса.
- `z` - нормированное отклонение.
- `file` - файл релиза.
- `hash` - SHA-256 контрольная сумма.

## 4. ЕДИНИЦЫ ИЗМЕРЕНИЯ

- Физические величины указывать в SI.
- Версии указывать в формате `MAJOR.MINOR.PATCH`.
- Даты указывать в ISO 8601.
- Хэши указывать как полные SHA-256 строки.

## 5. ПРИМЕРЫ РАСЧЕТОВ

Если отчет содержит `z = 0.7`, а контроль систематических ошибок не выявил устойчивого эффекта, релиз должен формулировать результат как совместимый с нулем.

Если два файла имеют разные SHA-256 после выгрузки в архив, публикацию остановить до выяснения причины.

## 6. ЧТО ИМЕННО ДОЛЖНА ПРОВЕРИТЬ ЛАБОРАТОРИЯ

### НАУЧНЫЙ КОНТРОЛЬ

- [ ] Указано, что антигравитация не доказана.
- [ ] Есть формулы, переменные, единицы и неопределенности.
- [ ] Указаны нулевые и отрицательные серии.
- [ ] Есть критерии фальсификации.
- [ ] Нет подмены инженерной компенсации веса изменением гравитации.

### ДАННЫЕ

- [ ] Нет raw data без review.
- [ ] Нет персональных данных.
- [ ] Нет закрытых параметров стенда.
- [ ] Есть словарь полей.
- [ ] Есть манифест файлов.
- [ ] Все хэши проверены.

### ПРАВА И ПУБЛИКАЦИЯ

- [ ] Лицензия указана.
- [ ] Авторство указано.
- [ ] GitHub commit указан.
- [ ] DOI/архивный URL указан после публикации.
- [ ] Патентный review не выявил блокеров.

## 7. КАКИЕ ДАННЫЕ НУЖНО СОБРАТЬ

- Название релиза.
- Версия релиза.
- Список файлов.
- SHA-256 каждого файла.
- Ответственный reviewer.
- Статус scientific review.
- Статус security review.
- Статус patent review.
- Статус license review.
- Итоговое решение: публиковать, отложить, закрыть как внутреннее, заменить.

## 8. КАКИЕ ВЫВОДЫ ДЕЛАТЬ НЕЛЬЗЯ

- Нельзя выпускать релиз с фразой "антигравитация доказана".
- Нельзя выпускать релиз без отрицательных результатов, если они есть.
- Нельзя публиковать технические режимы, которые могут быть патентоспособными.
- Нельзя публиковать данные без единиц и неопределенности.
- Нельзя считать GitHub push научной рецензией.

## 9. КАК ОТЛИЧАТЬ НАУЧНЫЙ РАСЧЕТ ОТ ПСЕВДОНАУЧНОГО ЗАЯВЛЕНИЯ

Научный релиз проходит checklist до публикации и допускает отрицательный итог. Псевдонаучный релиз заранее знает нужный вывод, а проверку использует как оформление.

## 10. КАКИЕ СЛЕДУЮЩИЕ ДОКУМЕНТЫ НУЖНО ПОДГОТОВИТЬ

- Таблицу подписей reviewer-ов.
- Журнал блокеров публикации.
- Шаблон release notes.
- Протокол отзыва или замены архивной версии.
- Юридический шаблон для внешнего DOI/архива.
